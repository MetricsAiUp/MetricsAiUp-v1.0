## ВЫСОКИЙ ПРИОРИТЕТ

### 1. Валидация входных данных на бэкенде
**Проблема:** POST/PUT маршруты принимают `req.body` напрямую без проверки. Можно отправить невалидные данные, сломать БД или вызвать ошибку.

**Что сделать:**
- Установить Zod (лёгкая библиотека валидации с TypeScript поддержкой)
- Создать схему для каждого эндпоинта: users (email формат, пароль мин. 6 символов), zones (name обязательно, type из enum), posts (zoneId существует), workOrders (orderNumber уникален)
- Middleware `validate(schema)` перед каждым POST/PUT
- Понятные ошибки на русском: "Поле email обязательно", "Пароль минимум 6 символов"

**Затрагивает:** `backend/src/routes/` — все 10 модулей маршрутов
**Оценка:** ~2-3 часа

---

### 2. Drag-and-drop планировщик заказ-нарядов
**Проблема:** Таймлайн DashboardPosts показывает ЗН на постах, но нельзя перемещать. Мастер-приёмщик планирует смену вручную, записывая на бумаге или в Excel.

**Что сделать:**
- Расширить DashboardPosts: блоки ЗН на таймлайне перетаскиваются мышью
- Drag горизонтально = изменение времени начала/конца
- Drag вертикально = перенос на другой пост
- Свободные ЗН из очереди внизу можно перетащить на пост
- При drop → POST /api/work-orders/:id/assign { postId, startTime, endTime }
- Визуальная проверка коллизий (два ЗН на одном посту в одно время → красная рамка)
- Кнопка "Сохранить расписание"

**Затрагивает:** `DashboardPosts.jsx`, `components/dashboardPosts/TimelineRow.jsx`, новый эндпоинт на бэкенде
**Оценка:** ~6-8 часов (сложная задача)

### 3. Двусторонняя синхронизация с 1С Альфа-Авто
**Проблема:** Данные из 1С загружаются вручную через Excel. Завершённые ЗН не возвращаются обратно в 1С. Двойной ввод данных.

**Что сделать:**
- **Автоматический импорт:** Бэкенд-сервис, который раз в N минут проверяет папку `/data/1c-import/` на новые .xlsx файлы и импортирует автоматически
- **Экспорт в 1С:** Кнопка "Выгрузить в 1С" → генерирует xlsx с завершёнными ЗН в формате 1С
- **REST API 1С (если есть):** Прямое подключение к HTTP-сервису 1С для push/pull данных
- **Лог синхронизации:** Таблица с историей: когда, сколько записей, ошибки

**Затрагивает:** `backend/src/routes/workOrders.js`, `Data1C.jsx`, новый сервис `backend/src/services/sync1C.js`
**Оценка:** ~4-6 часов (зависит от наличия API 1С)

### 4. Экспорт аналитики в Excel/PDF
**Проблема:** Страница Analytics показывает 6 графиков и таблицу сравнения, но нельзя скачать для отчёта директору или на совещание.

**Что сделать:**
- **Кнопка "Экспорт в Excel"** → xlsx файл с:
  - Лист 1: Сводные KPI (загрузка, эффективность, кол-во авто)
  - Лист 2: Данные по постам (таблица сравнения)
  - Лист 3: Ежедневная история (30 дней)
- **Кнопка "Экспорт в PDF"** → html2canvas + jsPDF:
  - Графики как изображения
  - Таблицы как текст
  - Шапка с логотипом и датой
- **Экспорт конкретного графика** → ПКМ на графике → "Скачать как PNG"
- Использовать xlsx (уже в проекте) + добавить jsPDF

**Затрагивает:** `Analytics.jsx`, новый утил `utils/export.js`
**Оценка:** ~3-4 часа

### 5. Управление сменами
**Проблема:** DashboardPosts имеет настройку начало/конец смены (08:00-20:00), но это просто UI-рамка. Нет реального управления: кто работает, какая смена, передача смен.

**Что сделать:**
- **Новая страница `/shifts`:**
  - Создание смен (утро 08:00-14:00, вечер 14:00-20:00, ночь 20:00-08:00)
  - Привязка работников к сменам: drag-and-drop или чекбоксы
  - Календарь смен на неделю/месяц
  - Статистика по сменам: выработка, кол-во ЗН, эффективность
- **Prisma модель:** Shift (id, name, startTime, endTime, date), ShiftWorker (shiftId, workerId)
- **Интеграция с DashboardPosts:** текущая смена подсвечивается, переключатель смены в header

**Затрагивает:** новая страница, новые модели в Prisma, новые API эндпоинты
**Оценка:** ~8-10 часов

## СРЕДНИЙ ПРИОРИТЕТ

### 6. Мобильная адаптивность
**Проблема:** Sidebar фиксирован 160px, таблицы не скроллятся горизонтально, модальные окна выходят за экран на мобильных устройствах. Мастера на СТО часто смотрят с планшетов.

**Что сделать:**
- **Sidebar:** на экранах < 768px → hamburger-меню (выезжает слева), на >= 768px — как сейчас
- **Таблицы:** горизонтальный скролл на мобильных, sticky первая колонка
- **Модальные окна:** max-width: 90vw, padding: 16px на мобильных
- **Карточки Dashboard:** 1 колонка на мобильных, 2 на планшетах, 4 на десктопе
- **Шрифты:** адаптивные размеры (clamp)
- **Touch-события:** свайп для закрытия sidebar/модалок

**Затрагивает:** `Layout.jsx`, `Sidebar.jsx`, все страницы с таблицами и модалками
**Оценка:** ~4-5 часов

### 7. Уведомления о критических событиях (real-time)
**Проблема:** Критические ситуации (авто без работы, клиент не приехал, пост простаивает) обнаруживаются только при ручном просмотре. Мастер может пропустить.

**Что сделать:**
- **Socket.IO → Toast:** При получении события типа `recommendation` → показать toast с звуком
- **Типы уведомлений:**
  - `no_show` (красный) — "Клиент Х не приехал на ЗН Y"
  - `vehicle_idle` (жёлтый) — "Авто АВ123 без работы 15 мин на посту 3"
  - `post_free` (синий) — "Пост 7 свободен более 30 мин"
  - `work_overtime` (оранжевый) — "ЗН 12345 превышает нормо-часы на 20%"
- **Центр уведомлений:** иконка колокольчика в header с бейджем, клик → dropdown с историей
- **Звук:** опциональный звуковой сигнал (можно отключить в настройках)
- **Настройки:** какие типы уведомлений показывать (чекбоксы в профиле)

**Затрагивает:** `Layout.jsx` (header), `hooks/useSocket.js`, новый компонент `NotificationCenter.jsx`
**Оценка:** ~4-5 часов

### 8. Переключатель слоёв на карте
**Проблема:** MapViewer показывает все элементы одновременно — при большом количестве элементов карта перегружена. Нельзя посмотреть только камеры или только посты.

**Что сделать:**
- **Панель слоёв** (правый верхний угол карты):
  - ☑ Посты (с цветом статуса)
  - ☑ Зоны (полупрозрачные области)
  - ☑ Камеры (с конусами FOV)
  - ☑ Проезды
  - ☑ Стены/двери
  - ☑ Подписи
  - ☑ Фоновое изображение
- Каждый слой можно включить/выключить чекбоксом
- Состояние сохраняется в localStorage

**Затрагивает:** `MapViewer.jsx`
**Оценка:** ~1-2 часа

### 9. Тепловая карта загрузки постов
**Проблема:** Analytics показывает общую загрузку за период, но не видно в какие часы какие посты перегружены, а какие простаивают. Мастер не может оптимизировать расписание.

**Что сделать:**
- **Heatmap-компонент:** таблица-сетка
  - Ось Y: посты (1-10)
  - Ось X: часы дня (08:00-20:00, по 30 мин)
  - Цвет ячейки: зелёный (свободен) → жёлтый (занят) → красный (перегружен)
  - Tooltip при наведении: "Пост 3, 10:00-10:30 — загрузка 85%, 2 ЗН"
- **Данные:** из `analytics-history.json` (hourly breakdown уже есть)
- **Фильтр:** по дню недели или конкретной дате
- Новый таб на странице Analytics или отдельный блок

**Затрагивает:** `Analytics.jsx`, новый компонент `Heatmap.jsx`
**Оценка:** ~3-4 часа

### 10. Экспорт карты в PNG/PDF
**Проблема:** MapViewer нельзя скачать для печати или отправки. Мастер хочет распечатать карту с текущими статусами.

**Что сделать:**
- **Кнопка "Скачать"** в toolbar MapViewer
- Konva `stage.toDataURL()` → скачивание PNG
- Опция PDF: canvas → jsPDF с заголовком "Карта СТО на {дата} {время}"
- Включать/не включать фон (чертёж) — чекбокс

**Затрагивает:** `MapViewer.jsx`
**Оценка:** ~1 час

### 11. Аудит-лог действий пользователей
**Проблема:** Нет записи кто какие данные менял. Если кто-то удалил пользователя или изменил настройки — нельзя узнать кто и когда.

**Что сделать:**
- **Prisma модель:** `AuditLog { id, userId, action, entity, entityId, oldData, newData, createdAt }`
- **Middleware:** перехватывает POST/PUT/DELETE, записывает в лог
- **Страница `/audit`** (только для admin): таблица с фильтрами по пользователю, действию, дате
- **Actions:** create_user, update_user, delete_zone, update_map_layout, acknowledge_recommendation и т.д.

**Затрагивает:** новая модель Prisma, middleware, новая страница
**Оценка:** ~3-4 часа

### 12. Snap-to-grid в редакторе карты
**Проблема:** Элементы перемещаются свободно — трудно выровнять посты ровно, зоны одинакового размера.

**Что сделать:**
- При перемещении элемент "прилипает" к ближайшей линии сетки (каждые 10px или 20px)
- При изменении размера — тоже snap
- Кнопка вкл/выкл snap (рядом с кнопкой сетки)
- Визуальные направляющие (guides): синие линии появляются когда элемент выровнен с другим

**Затрагивает:** `MapEditor.jsx` — обработчики onDragEnd, onTransformEnd
**Оценка:** ~1-2 часа

### 13. Undo/Redo в редакторе карты
**Проблема:** Если случайно удалил или переместил элемент — нельзя отменить. Приходится перезагружать.

**Что сделать:**
- Стек истории: массив снимков состояния (до 50 шагов)
- **Ctrl+Z** — отмена последнего действия
- **Ctrl+Shift+Z** (или Ctrl+Y) — повтор
- Кнопки ↩ ↪ в toolbar
- Записываются: перемещение, изменение размера, удаление, добавление, изменение свойств

**Затрагивает:** `MapEditor.jsx`
**Оценка:** ~2 часа

### 14. Дубликация элементов в редакторе
**Проблема:** Чтобы создать 10 одинаковых постов — нужно 10 раз кликать инструмент и настраивать каждый.

**Что сделать:**
- **Ctrl+D** или кнопка "Дублировать" в панели свойств
- Копия создаётся рядом с оригиналом (offset +20px, +20px)
- Имя автоинкрементируется: "ПОСТ 01" → "ПОСТ 01 (копия)"
- Можно выбрать несколько элементов (Shift+Click) и дублировать группу

**Затрагивает:** `MapEditor.jsx`
**Оценка:** ~1 час

### 15. Легенда на карте просмотра
**Проблема:** Пользователь не знает что означают цвета на карте — зелёный это свободно? Красный это проблема или камера?

**Что сделать:**
- Компактная легенда в правом нижнем углу:
  - 🟢 Свободен | 🟡 Занят | 🔴 Простой | 🟣 В работе
  - 🔴○ Камера | ▭ Зона | ⊡ Пост
- Сворачиваемая (кнопка показать/скрыть)
- Адаптивная: на мобильных — маленькая, на десктопе — полная

**Затрагивает:** `MapViewer.jsx`, новый компонент `MapLegend.jsx`
**Оценка:** ~1 час

## НИЗКИЙ ПРИОРИТЕТ

### 16. Предиктивная аналитика (ML)
**Идея:** На основе исторических данных (30+ дней) прогнозировать:
- Когда пост освободится (± 15 мин точность)
- Загрузку на завтра/неделю по часам
- Оптимальное расписание ЗН (минимизация простоев)
- Вероятность no-show для конкретных клиентов

**Реализация:** Python backend + scikit-learn, данные из Prisma, результаты в REST API
**Оценка:** ~20+ часов (data science)

---

### 17. Мультитенантность (несколько СТО)
**Идея:** Один инстанс для сети СТО (франшиза):
- Переключатель СТО в header
- `locationId` во всех моделях (User, Post, Zone, Session, WorkOrder)
- Общий дашборд для директора (сводка по всем СТО)
- Отдельные карты для каждого СТО

**Оценка:** ~15-20 часов (архитектурная переделка)

---

### 18. PWA + Push-уведомления
**Идея:** Установка как приложение на телефон без App Store:
- `manifest.json` + service worker
- Оффлайн-режим (кеширование страниц)
- Push-уведомления через Web Push API (новые ЗН, критические события)
- Иконка на домашнем экране

**Оценка:** ~4-5 часов

---

### 19. Таймер обратного отсчёта на посту
**Идея:** На карте и в DashboardPosts — обратный отсчёт до плановой сдачи ЗН:
- Зелёный: > 30 мин до дедлайна
- Жёлтый: 10-30 мин
- Красный: < 10 мин или просрочено
- Мигающий: просрочено > 15 мин
- Прямо на элементе поста на карте: "⏱ 01:23:45"

**Оценка:** ~2-3 часа

---

### 20. Фото/видео фиксация при приёмке
**Идея:** Мастер фотографирует повреждения авто при приёмке:
- Камера телефона через `getUserMedia()` или загрузка фото
- Привязка к ЗН и сессии авто
- Галерея фото в карточке сессии
- Хранение: `/data/photos/` или S3

**Оценка:** ~5-6 часов

---

### 21. QR-коды для автомобилей
**Идея:** При въезде автомобиля → генерируется QR-код:
- QR содержит: номер сессии, гос. номер, время въезда
- Механик сканирует QR → открывается карточка ЗН
- Печать QR на бумаге (прикрепляется к ключам)
- Библиотека: qrcode.js

**Оценка:** ~2-3 часа

---

### 22. Дашборд для мобильного механика
**Идея:** Упрощённый вид `/my-post` для механиков на планшетах:
- Только свой пост (определяется по привязке worker → post)
- Текущий ЗН с деталями (авто, тип работ, нормо-часы)
- Таймер работы
- Кнопки: "Начал работу" / "Завершил" / "Перерыв"
- Чек-лист операций по ЗН
- Никакой навигации — одна страница

**Оценка:** ~4-5 часов

---

### 23. Интеграция с Telegram
**Идея:** Telegram-бот для мастеров и директора:
- Уведомления: новый ЗН, no-show, простой, превышение нормо-часов
- Команды: `/status` — текущая загрузка, `/post 3` — инфо по посту
- `/report` — сводка за день
- Webhook: бэкенд → Telegram Bot API

**Оценка:** ~3-4 часа

---

### 24. Тёмная/светлая тема для карты
**Идея:** MapViewer сейчас одинаково выглядит в обеих темах:
- Тёмная тема: тёмный фон canvas, белые подписи, яркие элементы
- Светлая: белый фон, тёмные подписи
- Фон чертежа: инверсия или полупрозрачность в тёмной теме

**Оценка:** ~1-2 часа

---

### 25. Версионирование карт
**Идея:** История изменений layout карты:
- При каждом сохранении → новая версия (version: 1, 2, 3...)
- Список версий с датой и автором
- Кнопка "Откатить к версии N"
- Сравнение версий (diff элементов)
- Модель: `MapLayoutVersion { layoutId, version, elements, createdBy, createdAt }`

**Оценка:** ~3-4 часа
